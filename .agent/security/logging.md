# Logging

## Purpose

What may and may not be written to evidence logs, console output, or
committed files, so that quality evidence and debugging output never
become a secret- or personal-data-leak surface.

## Rules

- No token, session identifier, cookie, API key, or credential is ever
  written to: `quality/runtime/execution/latest/`, `quality/runtime/execution/runs/`,
  console output captured into evidence, commit messages, or handoff
  documents. See [`secrets.md`](secrets.md).
- No personal or customer data (per [`data-protection.md`](data-protection.md))
  is written into the same locations. Test fixtures and evidence runs use
  synthetic accounts/data only.
- Evidence generated by `npm run quality:evidence:fast` /
  `quality:evidence:full` (see [`../quality/evidence.md`](../quality/evidence.md))
  must be reviewed for accidental capture of the above before being
  treated as committable — the evidence tooling captures real command
  output, which can include stack traces or request/response bodies that
  need scrubbing if they touch auth flows.
- Console/debug logging added to application code must not print tokens,
  full user objects containing auth fields, or Supabase session payloads,
  even temporarily during development — remove debug logging that would
  leak these before the change is reported done.
- Large generated artifacts (recordings, full HTML reports, screenshots
  with potentially sensitive UI state) stay under
  `quality/runtime/execution/runs/`, which is gitignored per `../master.md`
  principle 14 — only lightweight `latest/` summaries are tracked, and
  those summaries must not carry secrets either.
- If a required evidence command's output would necessarily include
  sensitive data to be useful (rare), summarize and redact rather than
  omitting the evidence requirement entirely — see
  [`../quality/evidence.md`](../quality/evidence.md) for how gaps are
  recorded (`notAvailable`, never fabricated as passed).

## Review checklist

- Scan evidence/log output for secret-shaped strings before treating a run
  as complete.
- Confirm no personal/customer data appears in runtime or CI-artifact evidence
  summaries.
- Confirm debug logging touching auth/session state was removed before
  the task is reported done.
