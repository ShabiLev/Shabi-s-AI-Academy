# i18n

## Purpose

Explain how bilingual content actually flows through this codebase so new
strings are added correctly rather than hard-coded.

## Project-specific interpretation

`LanguageContext` holds the active language (`"he" | "en"`) and is the single
source pages read from — pages do not branch UI copy on their own
conditionals beyond selecting the right translation object, as seen in
`AgentsPage.tsx` (`const ui = language === "he" ? "he" : "en"`) and
`StarterAgentsPage.tsx`'s inline `text` object keyed by language. Several
domain modules ship their own bilingual data objects colocated with the
domain rather than one giant global dictionary — e.g.
`src/glossary/glossaryData.ts` rows carry `[id, termEn, termHe, defEn, defHe,
exampleEn, exampleHe]`, and `agentUi.ts` exports per-language UI-string
maps. Both patterns coexist; match whichever pattern the file you're editing
already uses.

## Constraints

- Every new user-facing string needs both a Hebrew and an English value —
  no string ships in only one language.
- Never concatenate translated fragments into a sentence; translate whole
  sentences with interpolated variables, per the standard.
- Direction (`dir`) is derived from the active language, not hard-coded per
  component.

## Known limitations

- Translation data is split across multiple colocated files
  (`i18n/translations.ts`, `glossary/glossaryData.ts`, per-feature `*Ui.ts`
  files) rather than one central catalog — there is no single "missing key"
  linter across all of them; parity is maintained by convention and by unit
  tests that check key presence where such tests exist.
- Only Hebrew and English are supported; the architecture doc's "future
  languages require typed catalog tooling" is explicitly not built.

## Current implementation status

Shipped: `LanguageContext`-driven bilingual UI across all shipped pages,
per-domain bilingual data files, glossary and Starter Catalog content fully
bilingual. Not shipped: a third language, a centralized translation-key
registry/linter.
