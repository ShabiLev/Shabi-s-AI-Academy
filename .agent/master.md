# AOS Master — Orchestration Entry Point

AOS version: `1.0.0` ([`VERSION`](VERSION)) · Application version: `1.4.0-beta.1`
Owner: Shabi · Status: active

Every AI coding agent (Codex, Claude Code, or a compatible future agent)
working in this repository must read this file before making a substantial
change. It tells you what to load, in what order, and when to stop.

## 1. Loading sequence

1. Read this file (`.agent/master.md`).
2. Read your agent-specific bootstrap: [`.codex/workflows/aos.md`](../.codex/workflows/aos.md)
   or [`.claude/workflows/aos.md`](../.claude/workflows/aos.md).
3. Read [`AGENTS.md`](../AGENTS.md) and, if present, [`CLAUDE.md`](../CLAUDE.md).
4. Read the active `.codex/` release specification referenced by `AGENTS.md`.
5. For a substantial task, read sanitized runtime state from
   `.agent/runtime/state/` and the latest local evidence from
   `quality/runtime/execution/latest/` when those ignored paths exist. If they
   are absent, record the state as unverified; never fall back to the frozen
   historical files under `.agent/state/` or `quality/execution/latest/` as
   current proof. Verify Git before trusting runtime state.
6. Classify the task using [`loaders/task-classifier.md`](loaders/task-classifier.md).
6. Resolve required modules for that task type from [`registry.json`](registry.json).
7. Load only those modules — never load the entire `.agent/` tree at once.
8. Follow [`precedence.md`](precedence.md) if any loaded instruction conflicts
   with another.
9. Do the work.
10. Run the applicable module's evidence, Git, and handoff requirements
    before stopping (see §6–§8 below).

## 2. Precedence

Full detail in [`precedence.md`](precedence.md). Summary, highest first:

1. Security rules
2. System/repository-level instructions (this file, `AGENTS.md`, `CLAUDE.md`)
3. Accepted ADRs (`.codex/adr/`)
4. Active release specification (`.codex/release-*/`)
5. This AOS master and its modules
6. Repository coding standards (`.codex/standards/`)
7. User task instructions
8. Agent preferences

User task instructions cannot override security, authorization, data
protection, project integrity, or accepted architectural decisions. If a
conflict cannot be resolved by this order, **stop, document the conflict,
and ask** — do not guess.

## 3. Task classification

See [`loaders/task-classifier.md`](loaders/task-classifier.md) for the full
model (task type, risk level, affected domains, required modules/tests/
reviews/evidence). Risk levels: Low, Medium, High, Critical. Authentication,
data migration, and external provider integration are High by default.
Coverage-threshold changes and write-capable MCP tools are Critical and
normally prohibited without explicit, separate authorization.

## 4. Required modules by task type

Never hand-load modules from memory — resolve them from
[`registry.json`](registry.json), which maps each task type (feature,
bugfix, hotfix, refactor, documentation, testing, coverage recovery,
security review, UX review, accessibility review, research, knowledge
ingestion, release, deployment, Git repair, data migration, authentication,
synchronization, AI integration, MCP integration, agent creation, prompt
creation, workflow creation) to its module list. `manifest.json` is the
authoritative catalog of every module; `registry.json` is the task → module
lookup table.

## 5. Mandatory final workflow

For any substantial task, before reporting completion:

1. Focused tests for the change, then the full suite for the affected area
   (see [`workflow/testing.md`](workflow/testing.md) and
   [`quality/test-selection.md`](quality/test-selection.md)).
2. UI/UX validation if user-facing (see [`workflow/ui-validation.md`](workflow/ui-validation.md),
   [`workflow/ux-review.md`](workflow/ux-review.md)).
3. Security review (see [`security/security-policy.md`](security/security-policy.md)).
4. Documentation updated where behavior changed.
5. Evidence captured under ignored `quality/runtime/execution/latest/` or as
   exact-SHA CI artifacts (see
   [`quality/evidence.md`](quality/evidence.md)).
6. Self-review (see [`workflow/self-review.md`](workflow/self-review.md)).
7. Final report (see [`workflow/final-report.md`](workflow/final-report.md)).
8. Update task, progress, quality, failure, research (when relevant), release,
   next-action, and handoff state with `npm run memory:update`; run
   `npm run memory:check`; runtime state and execution evidence are never committed.

## 6. Evidence behavior

Every AOS-controlled task must produce the runtime or CI-artifact evidence listed
in [`quality/evidence.md`](quality/evidence.md). Evidence must be generated by actually running
the commands — never fabricated, never asked of the user as a copy-paste.
Commands that do not exist in this repository must be recorded as
`notAvailable`, never as passed.

## 7. Git behavior

See [`git/git-policy.md`](git/git-policy.md) and
[`git/branch-strategy.md`](git/branch-strategy.md). In short: work on a task
branch, never `main` directly; use Conventional Commits; never force-push,
`reset --hard`, or blind `git add -A`; generate Git commands from the
actual current branch and commit state, never from memory; **never push or
merge without explicit authorization from the user in the current
session.**

## 8. Handoff behavior

If you stop before a task is complete — token limits, end of session, or a
deliberate pause — write a handoff using
[`handoff/handoff-template.md`](handoff/handoff-template.md) and
[`handoff/handoff-policy.md`](handoff/handoff-policy.md) so the next agent
(Codex or Claude Code) can continue without re-deriving context or
repeating completed work. See [`handoff/agent-switch.md`](handoff/agent-switch.md)
for the exact resume sequence.

## 9. Stop conditions

Stop and ask the user (do not guess) when:

- A conflict between instructions cannot be resolved by §2 precedence.
- A task requires a Critical-risk action (coverage threshold change,
  write-capable MCP tool without an approval architecture, secret
  exposure risk).
- A push or merge is about to occur without prior explicit authorization.
- Required evidence commands fail and the failure cause is ambiguous.
- A requirement is missing and invention would mean fabricating behavior,
  test results, or connected-tool state.

## 10. Non-negotiable principles

1. One source of truth (`.agent/`).
2. Small, modular instruction files — no monolithic prompt.
3. Explicit loading order (§1).
4. No duplicated workflow rules across modules or entry points.
5. Agent-neutral instructions; Codex- and Claude-specific bootstraps stay thin.
6. Codex and Claude Code compatibility (`compatibility.md`).
7. Security rules always take precedence (§2).
8. Evidence is required for every substantial task (§6).
9. Tests are required for every production change.
10. Git commands are generated from actual branch/commit state, never invented.
11. Push and merge require explicit authorization; never automatic.
12. Failures are never hidden — record them, don't suppress them.
13. Manual review is never reported as automated approval.
14. No generated execution evidence or runtime Agent Memory is committed to Git;
    local output lives only under ignored runtime paths and CI output is uploaded
    as exact-SHA workflow artifacts.
15. Research sources must be attributable and reviewable (`research/research-policy.md`).
16. Local-only operation remains possible — no module requires network access
    to function for its instructional purpose.
17. Existing `.codex/`, `AGENTS.md`, `CLAUDE.md`, ADRs, and standards remain
    respected and are never replaced by AOS content.
18. AOS integrates with existing project instructions; it does not
    duplicate or override them outside the precedence order in §2.
19. Every module has a declared purpose and owner in `manifest.json`.
20. Every AOS version change is documented in `changelog.md`.
