# Internationalization Architecture

## Purpose

Make Hebrew RTL and English LTR equally complete.

## Current state

A language context selects translations and sets document language/direction. Layout uses logical CSS properties and semantic order.

## Decision and constraints

Every user-facing string has both translations. Direction comes from the active language; data content may declare its own language. Dates and numbers use locale-aware formatting.

## Dependency boundaries

Components consume translation keys, not language-conditionals for copy. CSS controls mirroring without duplicate component trees.

## Anti-patterns

Hard-coded strings, left/right spacing for logical layouts, reversing non-directional icons, or using RTL as visual-only alignment.

## Testing impact

Unit-test translation key presence; Playwright and axe cover both directions, overflow, keyboard order, and mobile wrapping.

## Future evolution

Future languages require typed catalog tooling and locale-aware search decisions.
