# Observability

## Purpose

State what can and can't actually be observed in this local-first app, so
an agent doesn't assume a telemetry/logging backend exists to instrument
against.

## Authoritative source(s)

- [.codex/architecture/performance.md](../../.codex/architecture/performance.md)
  ("keep normal workflows network-independent")
- [.codex/architecture/security.md](../../.codex/architecture/security.md)
  ("logging sensitive input" is a forbidden practice)
- `src/pages/AnalyticsPage.tsx`, `src/workspace/` (`WorkspaceContext`,
  `workspaceStorage.ts`)
- `.agent/quality/evidence.md`, `.agent/security/logging.md`

## Project-specific interpretation

There is no third-party telemetry, analytics SaaS, or remote logging
service anywhere in this app — `package.json` has no Sentry/Datadog/GA-style
dependency. `AnalyticsPage.tsx` is local-only: it reads
`state.analytics` events (`routeViewed`, `searchPerformed`,
`commandExecuted`, `assistantIntent`, `workflowRun`) that `WorkspaceContext`
already recorded to local storage, lets the learner filter by date range,
and offers a client-side JSON export — nothing leaves the browser.
Runtime execution observability is the bounded, sanitized run history
described in `.agent/knowledge/storage.md` (`RunEvent.safeSummary`,
capped at `MAX_RUNTIME_RUNS`), not a logging pipeline.

For the *coding agent's own work* (not the product's runtime), the AOS
evidence system (`quality/runtime/execution/latest/`, produced by
`scripts/run-quality-evidence.mjs`) is the substitute for runtime telemetry:
it is how a change's test/build/lint results are captured as reviewable
artifacts, per `.agent/quality/evidence.md`.

## Constraints

- Never add a remote analytics/telemetry SDK to satisfy an "observability"
  request without treating that as a new integration decision (new
  dependency, new network dependency, new privacy surface) — flag it,
  don't silently wire it in.
- Runtime/event logs must use `safeSummary`-style sanitized text, never
  raw user input or anything resembling a credential, consistent with the
  security standard's "redacted logs" guidance and `.agent/security/logging.md`.
- AOS evidence must be generated by actually running the commands — never
  fabricated — per `master.md` §6.

## Known limitations

- Local-only analytics means there is no cross-user or aggregate view of
  app usage; each learner only ever sees their own browser's event history.
- There is no error-tracking/crash-reporting mechanism; a runtime exception
  outside the modeled `RuntimeError` union would surface only in the
  browser console, not to any dashboard.

## Current implementation status

Shipped: local, exportable workspace analytics; bounded, sanitized runtime
run history; the AOS evidence pipeline for agent-driven changes. Not
shipped: any remote telemetry, error-tracking, or logging backend.
