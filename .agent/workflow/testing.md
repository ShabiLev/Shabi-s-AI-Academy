# Testing

## Purpose

Define focused-vs-full test selection and the required npm commands per
change type, so evidence is generated by actually running the right tests —
not a generic "write tests" instruction. This module is phases 9–10 of
[`development.md`](development.md) and is required for `feature`, `bugfix`,
`hotfix`, `refactor`, `testing`, and `coverage recovery` task types.

## When to load

Load whenever code changes, and always before
[`../quality/evidence.md`](../quality/evidence.md) is run, since evidence
capture assumes these commands have already been exercised at least once
during the task.

## Prerequisites

- Implementation (or a bug reproduction) exists to test.
- [`../quality/test-selection.md`](../quality/test-selection.md) has been
  consulted for the task's specific risk level and domain.

## Required actions

1. **Focused first.** Run the narrowest command that exercises the change:
   a single Vitest file (`npx vitest run <path>`), or, for evidence-tooling
   changes, `npm run test:evidence` (the `node --test
   scripts/evidence-utils.test.mjs` suite).
2. **Unit/domain suite.** Run `npm run test:run` for the full Vitest suite.
   Use `npm run test:coverage` when the change is release-scale or touches
   coverage-sensitive code, per
   [`../quality/coverage.md`](../quality/coverage.md).
3. **Build check.** Run `npm run build` — a failing build is always a
   blocker, never a warning.
4. **E2E for user-visible flows.** Run `npm run test:e2e` (fast, Desktop
   Chromium only) for routine changes; run `npm run test:e2e:full` for
   release-scale or `High`/`Critical`-risk changes that touch navigation,
   forms, or protected routes.
5. **Journeys and UX suites when user-facing.** Run `npm run test:journeys`
   for task-completion flows and `npm run test:ux` for broader UX coverage;
   see [`ux-review.md`](ux-review.md) for when these are mandatory.
6. **Accessibility.** Run `npm run test:a11y` for any new or changed complex
   screen/state, per [`accessibility.md`](accessibility.md).
7. **Visual regression.** Run `npm run test:visual` when layout, spacing, or
   themed surfaces changed; only update snapshots via
   `npm run test:visual:update` after an intentional, reviewed visual
   change (see [`../quality/visual-regression.md`](../quality/visual-regression.md)).
8. **Performance.** Run `npm run test:performance` (Lighthouse
   collect+assert) for changes affecting primary routes or bundle size, per
   [`performance.md`](performance.md).
9. **Pages build parity.** Run `npm run build:pages` and
   `npm run test:e2e:pages` when the change could affect the GitHub Pages
   base-path/hash-router deployment, per [`deployment.md`](deployment.md).
10. **Release-scale gate.** For `release` task type or any `High`/`Critical`
    change, run `npm run validate:release` (the full gate combining lint,
    unit, coverage, build, catalog, docs, E2E, journeys, UX, a11y, visual,
    performance, and `git diff --check`).
11. Record every command actually run — pass, fail, or `notAvailable` — feed
    the results into [`../quality/evidence.md`](../quality/evidence.md); do
    not summarize from memory.

## Prohibited actions

- Arbitrary waits (`waitForTimeout`), brittle CSS selectors, network
  reliance, weakened expectations, blanket retries, or skipped defects —
  all forbidden by [`.codex/standards/testing.md`](../../.codex/standards/testing.md).
- Marking a flaky test as quarantined instead of reproducing and fixing the
  nondeterminism, per [`.codex/standards/qa.md`](../../.codex/standards/qa.md).
- Reporting a test command as `passed` when it was not actually run.
- Lowering a coverage threshold or disabling axe checks to make a run green
  — a `Critical`-risk action per
  [`../loaders/task-classifier.md`](../loaders/task-classifier.md).
- Closing a production defect without the smallest reliable regression test
  that would have caught it.

## Deliverables

- Console/log output for every command run in this task, retained long
  enough to populate evidence.
- New or updated test files for the change, following the test pyramid in
  [`.codex/standards/qa.md`](../../.codex/standards/qa.md).

## Evidence requirements

Every command run under this module must be reflected in
`quality/execution/latest/` via `npm run quality:evidence` (or the `:full`,
`:pages`, `:headed` variant appropriate to the task) — see
[`../quality/evidence.md`](../quality/evidence.md). A command that doesn't
exist in this repository (do not invent script names beyond
`package.json`'s `scripts` section) is recorded as `notAvailable`, never as
passed.

## Exit criteria

The focused command and the full command for the affected area both ran and
their real results (pass/fail/notAvailable) are recorded; any failure is
either fixed, or explicitly triaged and reported — never hidden.
